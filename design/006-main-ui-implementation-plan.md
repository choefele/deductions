# Main UI Implementation Plan

## Purpose

Implement the first real Deductions application shell based on `005-main-ui-plan.md`.

This implementation slice should prove the navigation model and main UI structure with mock data. It should not implement invoice parsing, persistence, PDF rendering, AI extraction, source integrations, or export generation.

## Product Decisions

- The app opens to an all-years dashboard.
- The layout has a left sidebar and a main content inset.
- The main header belongs inside the main content inset, matching the `sidebar-07` structure.
- Use Tailwind CSS and shadcn/ui components in the app.
- Use the Radix UI variant of shadcn/ui for the first implementation.
- Treat `app/renderer/components/ui/` as canonical shadcn-generated component code.
- Use React Router in Data Mode with hash-based routing for the Electron renderer.
- Sidebar sections are Profile, Review, Tax years, and Sources.
- Review contains open todos only.
- Tax years contain current decisions, including `Not tax-relevant`.
- `Needs consultant review` is a flag, not a status.
- Tax categories use English labels for now.
- Export is a tax-year dashboard action, not a persistent package section.
- Invoice lists live in the main panel, not the sidebar.
- Invoice detail opens in the main content area for the first slice.
- The quick review panel is deferred to a separate design document.

## Technical Approach

Use the existing React/Electron renderer with Tailwind CSS, shadcn/ui, and React Router.

The design should use shadcn's sidebar pattern structurally and use copied shadcn components directly in the app. The first slice should install only the components needed by this UI instead of adding a broad component set.

`app/renderer/components/ui/` is reserved for components generated by `shadcn add`. Do not put app-specific behavior or hand-written component forks in that directory. Treat it like vendor-like source owned by shadcn generation. If a generated component needs different product behavior, compose or wrap it from a file outside `components/ui`, such as `components/AppSidebar.tsx`, `components/MainHeader.tsx`, or a future `components/common/` folder. Manual edits to `components/ui` should be rare, deliberate, and documented as a fork.

Use React Router for real navigation from the start. Keep data mocked in the renderer for this slice; route loaders can read local mock data now and later be adapted to IPC-backed or persistence-backed data.

Mock data must be treated as a temporary data source behind a small data-access boundary. Views and route loaders should not import mock arrays directly. They should call repository-style functions such as `listTaxYears`, `getTaxYearSummary`, `listInvoicesByCategory`, `listReviewQueueInvoices`, and `getInvoiceById`. The first implementation can back those functions with in-memory mock data; a later implementation can replace the backing store with a local database without changing most UI components.

## UI Stack Decisions

### Tailwind And shadcn/ui

Set up Tailwind CSS for the Vite renderer and initialize shadcn/ui with project-local component output.

Recommended setup:

- Tailwind CSS v4 through `@tailwindcss/vite`.
- shadcn `new-york` style.
- CSS variables enabled.
- `zinc` or `neutral` base color.
- `lucide-react` icons.
- Import alias rooted at `app/renderer`, for example `@/* -> app/renderer/*`.

Initial shadcn components:

- `sidebar`
- `button`
- `breadcrumb`
- `separator`
- `avatar`
- `badge`
- `table`
- `card`
- `dropdown-menu`
- `collapsible`
- `tooltip`
- `scroll-area`
- `input`
- `skeleton`

Add more components only when a view actually needs them.

### Radix UI vs Base UI

Use Radix UI as the shadcn primitive variant for the first implementation.

Reasons:

- shadcn's sidebar docs still present the Radix variant as the default page, with Base UI available as an alternate variant.
- Radix-backed shadcn components are the more established path and have the broadest existing examples.
- This app needs common desktop-product primitives now: sidebar, dropdown menu, collapsible sections, tooltip, table, and later dialog/sheet behavior.
- The team can switch individual copied components later if Base UI becomes more attractive, because shadcn components live in the codebase rather than only inside `node_modules`.

Base UI remains worth watching, especially for newer shadcn variants and future component work. It is not the conservative first choice for this project unless we want to optimize for newer primitives over established examples.

### React Router Mode

Use React Router Data Mode with `createHashRouter`.

Reasons:

- Declarative Mode is enough for basic UI state, but Data Mode gives route-level loaders, actions, pending states, and `useFetcher` without adopting the full React Router framework.
- Framework Mode adds a Vite plugin, route modules, automatic code splitting, and SSR/static-rendering features that are not needed for an Electron renderer.
- Hash routing is pragmatic for Electron packaged builds because it avoids relying on server/path fallback behavior when the renderer is loaded from `file://`.
- Data Mode gives a clean migration path from mock route data to local persistence or IPC-backed loading.

Avoid `BrowserRouter` for the first slice. It is cleaner-looking in URLs, but it is less useful in a desktop Electron app and can create packaged-build edge cases. Avoid Framework Mode until the app has enough route/data complexity to justify its conventions.

```text
    Breadcrumbs.tsx

  views/
    AllYearsDashboard.tsx
    TaxYearDashboard.tsx
    CategoryView.tsx
    InvoiceDetailView.tsx
    ReviewQueueView.tsx
    ImportResultView.tsx
    SourcesView.tsx
```

Keep these files under `app/renderer/` because this is renderer-only UI work. shadcn-generated primitives should live under `app/renderer/components/ui/`.

Component ownership rules:

- `app/renderer/components/ui/`: canonical `shadcn add` output only.
- `app/renderer/components/`: app-specific composed components built from shadcn primitives.
- `app/renderer/components/common/`: optional future home for reusable app components.
- `app/renderer/views/`: route-level screens.
- `app/renderer/data/`: repository interfaces and adapters.

The repository interface should be renderer-local for the mock slice, but its shape should anticipate a later split:

- UI and route loaders call a repository interface.
- The mock repository returns data from in-memory fixtures.
- A later database repository can call preload IPC methods.
- The main process can own SQLite or another local database implementation.
- Shared types can move to `app/shared/` once they become IPC contracts.

## Mock Data Model

Start with typed mock data that resembles the future domain model closely enough to exercise UI behavior.

Do not let mock data shape the app accidentally. Keep domain types and repository functions separate from the fixture arrays so real local storage can replace the fixtures later.

Core types:

```ts
type ReviewStatus = 'pending' | 'accepted' | 'rejected';

type ReviewFlag = 'needs-consultant-review' | 'export-issue';

type TaxCategoryId =
  | 'work-related-expenses'
  | 'special-expenses'
  | 'extraordinary-burdens'
  | 'household-services'
  | 'tradesperson-services'
  | 'not-tax-relevant';

type Invoice = {
  id: string;
  taxYear: number;
  categoryId: TaxCategoryId;
  vendor: string;
  date: string;
  amount: number;
  currency: 'EUR';
  source: 'manual-upload' | 'connected-source';
  status: ReviewStatus;
  flags: ReviewFlag[];
  confidence: 'high' | 'medium' | 'low';
  note?: string;
};
```

Mock enough invoices to cover:

- Multiple tax years.
- Each first-slice category.
- Pending, accepted, and rejected statuses.
- Low-confidence invoices and consultant-review flags.
- Not-tax-relevant decisions.
- Empty states for at least one category or source.

## Data Access Boundary

Define a narrow repository interface before adding route loaders.

Example:

```ts
type DeductionRepository = {
  listTaxYears(): Promise<TaxYearSummary[]>;
  getTaxYearSummary(year: number): Promise<TaxYearSummary | null>;
  listInvoicesByCategory(
    year: number,
    categoryId: TaxCategoryId,
  ): Promise<Invoice[]>;
  listReviewQueueInvoices(queue: ReviewQueueId): Promise<Invoice[]>;
  getInvoiceById(invoiceId: string): Promise<Invoice | null>;
};
```

For the first slice:

- `mockDeductionRepository` implements this interface using mock fixtures.
- React Router loaders call repository functions.
- Components receive loader data and do not know whether it came from fixtures, IPC, or a database.
- Repository functions should return promises even when backed by synchronous mock data, because the future local database path will be asynchronous.

Future local database path:

1. Define persistent schema in a later storage design document.
2. Move stable domain types and repository request/response types to `app/shared/`.
3. Add preload methods for read operations and later write operations.
4. Implement the repository in the renderer as an IPC-backed adapter.
5. Implement database access in the main process.
6. Keep route loaders and views calling the same repository methods.

## Navigation State

Represent screens as React Router routes. Keep a small selection helper type for derived UI state, breadcrumbs, and tests.

```ts
type AppSelection =
  | { type: 'all-years' }
  | { type: 'tax-year'; year: number }
  | { type: 'category'; year: number; categoryId: TaxCategoryId }
  | { type: 'invoice'; invoiceId: string }
  | { type: 'review-queue'; queue: 'pending' | 'low-confidence' | 'consultant-review' | 'export-issues' }
  | { type: 'sources' }
  | { type: 'import-result' };
```

Selection changes should drive:

- Active sidebar item.
- Main view rendering.
- Breadcrumb labels.
- Table filters.
- Current invoice context.

Proposed route shape:

```text
/
/years/:year
/years/:year/categories/:categoryId
/invoices/:invoiceId
/review/pending
/review/low-confidence
/review/consultant-review
/review/export-issues
/sources
/import-result
```

Use hash URLs in the packaged app, for example `#/years/2025/categories/work-related-expenses`.

## Component Responsibilities

### `App`

- Creates or renders the router provider.
- Keeps the existing preload API available to routed views.
- Provides or imports the active repository adapter used by route loaders.

### `AppShell`

- Owns the visual two-panel layout.
- Contains `Sidebar` and main inset.
- Keeps the main header inside the inset.

### `Sidebar`

- Renders compact profile.
- Renders Review queues with counts.
- Renders Tax years with category children.
- Renders Sources.
- Supports expanded/collapsed state.
- Does not render full invoice lists.

### `MainHeader`

- Renders sidebar toggle.
- Renders breadcrumbs.
- Renders `Import invoice`.
- Renders disabled or placeholder search if search is not implemented.
- Uses shadcn `Breadcrumb`, `Button`, `Separator`, and later `Input`.

### `AllYearsDashboard`

- Shows tax-year summaries.
- Shows open todo counts.
- Shows recent imports or recent invoices.
- Links into tax-year dashboards and review queues.

### `TaxYearDashboard`

- Shows status counts and flag counts for one year.
- Shows category summaries.
- Shows export readiness.
- Provides `Export` as a disabled or placeholder action for this slice.

### `CategoryView`

- Shows a table of invoices for one tax year/category.
- Supports local status and flag filters if simple to add.
- Row click opens invoice detail.

### `ReviewQueueView`

- Shows a filtered invoice table for one todo queue.
- Row click opens invoice detail.

### `InvoiceDetailView`

- Shows placeholder document viewer.
- Shows extracted-fact placeholders and review metadata from mock data.
- Shows status and flags clearly.
- Does not implement editing yet unless it is trivial local state.

### `ImportResultView`

- Shows selected file count from the native file picker.
- Shows placeholder accepted/skipped/failed grouping.
- Links to pending review.

## Styling Plan

Use Tailwind utilities for component layout and use `styles.css` for global theme tokens, Tailwind imports, and any app-level primitives that should not live inline.

Layout requirements:

- Full-height app shell.
- Sidebar width around 300 px when expanded.
- Icon rail width around 56 px when collapsed.
- Main header height around 56 to 64 px.
- Sidebar content scrolls independently.
- Main content scrolls independently.
- Tables remain readable at desktop sizes.
- Narrow windows use an off-canvas-like sidebar behavior or a simple overlay fallback.

Visual requirements:

- Work-tool density, not landing-page styling.
- Neutral background and clear borders.
- Count badges in sidebar.
- Status and flag badges in tables.
- Stable dimensions for sidebar rows, buttons, and table cells.
- No decorative nested cards.

## Implementation Steps

### Step 0: UI Tooling And Router Setup

- Install Tailwind CSS v4 and `@tailwindcss/vite`.
- Configure the renderer Vite build to use the Tailwind plugin.
- Configure TypeScript and Vite aliases for `@/* -> app/renderer/*`.
- Initialize shadcn/ui with the Radix UI variant.
- Add initial shadcn components listed above.
- Do not hand-edit generated files in `app/renderer/components/ui/` during normal feature work.
- Install React Router.
- Add `router.tsx` using `createHashRouter` and `RouterProvider`.

### Step 1: Mock Domain Data

- Add `mockData.ts`.
- Define category, invoice, status, and flag types.
- Add `data/deductionRepository.ts` for the repository interface.
- Add `data/mockDeductionRepository.ts` for the fixture-backed implementation.
- Add helper functions for summaries, counts, and filters behind repository methods.
- Add unit tests for summary/filter helpers.

### Step 2: Navigation Model

- Add `navigation.ts`.
- Define route/selection helpers.
- Add breadcrumb derivation.
- Add active sidebar matching helpers.
- Add `routeData.ts` loaders that call repository methods instead of importing fixture arrays directly.
- Add unit tests for breadcrumbs and active states.

### Step 3: Shell Layout

- Replace the minimal setup panel with `AppShell`.
- Add shadcn `SidebarProvider`, `Sidebar`, `SidebarInset`, and main header.
- Keep the existing preload API and import dialog available.
- Update smoke test expectations from the old setup UI to the new shell.

### Step 4: Sidebar

- Add Profile, Review, Tax years, and Sources sections.
- Add count badges from mock summaries.
- Use shadcn `Sidebar`, `Collapsible`, `Badge`, `Avatar`, and `Tooltip` where appropriate.
- Ensure invoices are not listed as canonical sidebar children.

### Step 5: Dashboards

- Add all-years dashboard as the default view.
- Add tax-year dashboard with category summaries and placeholder export action.
- Link dashboard cards/rows to the right selections.

### Step 6: Tables And Detail

- Add reusable `InvoiceTable`.
- Add category and review queue views.
- Add invoice detail view with placeholder document panel.
- Ensure row links navigate to invoice routes and update breadcrumbs and active sidebar context.

### Step 7: Import Result Flow

- Keep `Import invoice` wired to `window.deductions.openFiles()`.
- After file selection, navigate to `#/import-result` and show `ImportResultView` with selected file count.
- Do not create real invoice records from selected files yet.

### Step 8: Polish And Responsive Pass

- Check expanded/collapsed sidebar behavior.
- Check narrow viewport behavior.
- Check table overflow.
- Check keyboard focus states.
- Check empty states.

## Acceptance Criteria

- Opening the app shows the all-years dashboard.
- The UI uses the finalized navigation model: Profile, Review, Tax years, Sources.
- Tailwind and shadcn/ui components are used for the shell and core controls.
- React Router Data Mode with hash routing drives main navigation.
- Review queues show only open todos.
- Tax years show fixed English tax categories including `Not tax-relevant`.
- Individual invoices are selected from main-panel lists, not from the sidebar.
- Invoice detail can be reached from a category table and a review queue.
- `Needs consultant review` is displayed as a flag.
- Export appears as a tax-year dashboard action.
- Import uses the existing native file picker and shows an import-result placeholder.
- Existing security constraints remain intact: renderer does not import Electron or Node APIs.
- Mock data is isolated behind repository functions so it can later be replaced by an IPC/database-backed repository.
- Unit tests and Electron smoke test are updated and passing.

## Risks

- The mock data model may accidentally become treated as the real domain model. Keep it clearly named as mock data.
- If views import mock fixtures directly, later database migration will be unnecessarily expensive. Enforce repository-only access from route loaders and views.
- Plain CSS may drift from the shadcn-inspired target if the layout is not checked visually.
- Without real document rendering, invoice detail may feel too abstract. The placeholder should still reserve realistic space and controls.
- Review queues and tax-year categories can show the same invoice through different routes; active-state logic must make that understandable.
